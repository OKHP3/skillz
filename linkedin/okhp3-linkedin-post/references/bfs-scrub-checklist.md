# Employer-Reference Scrub Checklist

Final gate before any LinkedIn output. Generic by design — this file should never itself contain employer-specific names, codenames, or identifying details. If you're tempted to add a specific example here, don't; describe the category instead.

Check the draft for:

- [ ] Employer name, in any form (full name, common abbreviation, informal nickname)
- [ ] Internal project or initiative codenames — anything that wouldn't mean anything outside the organization
- [ ] Proprietary system, product, platform, or tool names specific to the employer (internal or vendor-licensed-but-internally-branded)
- [ ] Internal workflow/ticketing tool references (system names, not generic categories — "our ticketing system" is fine, a specific product name tied to this employer is not)
- [ ] Specific organizational structure details beyond a generic role title — team names, reporting-line specifics, internal-only job titles
- [ ] Internal document/deck naming schemes or identifiers
- [ ] Combinations that identify by inference — no single item above triggers, but together they'd let a reader figure out the employer

## If something is caught

Don't just delete the sentence if the underlying INSIGHT is still valid and employer-agnostic — generalize it instead. "We migrated [Proprietary System] to [Other Proprietary System]" becomes "migrated a legacy on-prem system to a cloud platform" if the post's point is about the migration process, not the specific systems.

If the angle's core insight CANNOT survive generalization (the insight only matters with the specific identifying details), the angle itself may not be salvageable for LinkedIn. Flag this back to the user rather than publishing a hollowed-out version.

## Status logging

If this gate catches something, log it (category only, not content) so `okhp3-linkedin-angles` can learn — angles whose evidence is too employer-specific to generalize are worth flagging earlier, at triage, not at drafting.
