# SOP Structure Rules

## ISO 9001 §7.5 Requirements

Under ISO 9001:2015 §7.5 (Documented information), SOPs must:
- Be available and suitable for use, where and when needed (§7.5.3a)
- Be adequately protected from loss of confidentiality, improper use, or loss of integrity (§7.5.3b)
- Have a controlled version identity: document ID, version number, date, and approval status

## Mandatory SOP Sections

All SOPs generated by `generate-sop.mjs` must include:

| Section | Source in PNS | ISO reference |
|---|---|---|
| Header (document ID, version, date, owner) | PNS frontmatter | §7.5.2 |
| Purpose | `babok_core_concepts.need` + `process_box.trigger` | §4.4.1 |
| Scope | `process_box.inputs[]` + `process_box.outputs[]` | §4.4.1 |
| Definitions | All role names, system names, business rule terms | §7.5 |
| Responsibilities (RACI summary) | `roles_and_raci.raci_matrix[]` | §5.3 |
| Procedure steps | `activity_sequence.activities[]` | §4.4.2 |
| Exception handling | `exception_paths[]` | §10.2 |
| Business rules | `business_rules[]` | §6.1 |
| Related documents | PIR ref, PNS ref, BPMN ref | §7.5.3 |
| Revision history | `revision_history[]` | §7.5.2 |

## Procedure Step Writing Rules

Each procedure step must:
1. Begin with an imperative verb (not "The requester should submit…" → "Submit the purchase request…")
2. State one action only — no compound steps joined with "and" or "then"
3. Identify the actor (role, not person name) in the step header
4. Specify the input needed and output produced
5. Reference the system used (if any)
6. Note any decision criteria inline (e.g. "If amount > $1,000, continue to step 5; otherwise go to step 7")
7. Note exception handling inline (e.g. "If vendor quote is unavailable, see Exception EX-002")

## Work Instruction Generation Rules

Work instructions are scoped to a single role. For each role in `roles_and_raci.roles[]`:
- Extract only the activities where that role has R (Responsible) in the RACI matrix
- Include decision points where the role must make a judgement call
- Include exception paths owned by the role (`owner_role_id` matches)
- Do not include activities where the role is only C (Consulted) or I (Informed)

## Document Numbering Convention

Document IDs follow the pattern: `<org-prefix>-SOP-<process_id>-<version>`

Example: `OKHP3-SOP-proc-purchase-approval-v1.0`

If `org-prefix` is unknown, use `ORG-SOP-<process_id>-<version>` and note it as a placeholder.

## Revision History Rules

- Version `1.0` is the first approved version; drafts use `0.x` numbering
- Each revision entry requires: `version`, `date` (ISO 8601), `author_role` (not person name), `summary` (≤200 chars)
- Do not remove revision history entries — all versions must remain traceable
